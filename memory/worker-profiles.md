# Worker Profiles — эффективные промпты по ролям

> Обновляй когда находишь формулировку которая стабильно даёт 0 rework для роли.

## coder
- Всегда указывай: стиль проекта (tab/space, naming convention), конкретные файлы для изменения
- "Не добавляй зависимости без указания в результате" — предотвращает сюрпризы
- Для API: указывай точный формат ответа (HTTP code + body structure)
- Алгоритмы: описывай пошагово в промпте (прогон #3 analyzer: 5 шагов кластеризации → 0 rework)
- Если воркер импортирует из чужого модуля — указывай точные сигнатуры (dataclass поля, функции) прямо в промпте, не надеясь что файл будет к моменту запуска
- Конкретные имена тестов в промпте помогают: воркер пишет нужные + добавляет бонусные (11 из 8 заданных у reader, 13 из 6 у analyzer)

## tester
- "Покрой минимум 3 edge cases: пустой input, невалидный input, граничное значение"
- "Приложи полный stdout запуска тестов" — без этого невозможно ревьюить
- Указывай тестовый фреймворк явно (jest, pytest, etc.)

## researcher
- "Структура: Факты (что нашёл) → Выводы (что это значит) → Рекомендации (что делать)"
- "Максимум 1 страница, не больше" — без ограничения пишут простыни
- Указывай конкретные вопросы, не "исследуй тему X"

## writer
- Обязательно: целевая аудитория + тон + длина + формат
- "Пример хорошего текста в этом стиле: [ссылка/цитата]" — калибрует качество
- Для постов: "Первое предложение = hook, последнее = call to action"

## content_analyst
- "Извлеки: ключевые идеи (пронумеруй), сильные цитаты (дословно), метаданные"
- "Не интерпретируй, только извлекай" — на этапе extraction не нужен креатив
- Указывай формат inbox-файлов (chat export, markdown notes, etc.)

## architect
- "Диаграмма (mermaid) обязательна" — без визуализации архитектура не ревьюится
- "Trade-offs таблицей: вариант | плюсы | минусы | рекомендация"
- Ограничивай scope: "архитектура модуля X", не "архитектура всего проекта"
