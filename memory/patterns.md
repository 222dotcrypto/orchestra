# Patterns — что работает

> Обновляй после каждой задачи с rework=0. Один инсайт = одна строка. Удаляй устаревшее.

## Декомпозиция
- Задачи с бинарными acceptance criteria (да/нет) получают 0 rework чаще всего
- Top-down декомпозиция с проверкой "могу ли написать criteria?" — лучший тест готовности задачи
- Одна задача = одно главное действие. Два действия = две задачи

## Промпты воркеров
- 5C формула (Context, Command, Constraints, Criteria, Completion) — базовый шаблон
- Constraints "НЕ трогай файлы вне owned_files" предотвращает побочные эффекты
- Пример ожидаемого результата в description снижает rework на ~50%

## Координация
- Non-overlapping file ownership предотвращает конфликты
- Волновой запуск (все независимые → ревью → следующая волна) проще чем произвольный DAG
- Interface-файлы между фазами (сигнатуры, форматы) лучше чем передача полного результата

## Ревью
- Ревью < 30 сек = хорошо сформулированные criteria
- Rework-комментарий должен быть конкретнее чем criteria (пример + что именно не так)

## Ресурсы
- Trivial задачи (1 файл, <20 строк) — всегда DIY, воркер дороже
- Один хорошо написанный промпт > два быстрых + rework

## Инфраструктура (v2 — после анализа прогона #1)
- Протокол в файле (.brain/WORKER_PROTOCOL.md) → промпт 5 строк вместо 50. Воркер читает сам при старте
- Сигнальные файлы (.brain/signals/task-XXX.review) → Мозг проверяет одним `ls` вместо чтения N JSON-файлов
- Adaptive polling (3 мин пауза, потом каждые 60 сек) → ~80% меньше циклов мониторинга
- check-workers.sh (tmux/ps) заменяет heartbeat → убирает ~300 tool calls с воркеров за сессию
- --dangerously-skip-permissions в run-worker.sh → воркеры могут запускать тесты и любые команды
- Строгие фазы: не запускать зависимых воркеров до завершения предыдущей фазы → нет idle polling
