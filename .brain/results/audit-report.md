# Аудит Orchestra -- отчёт

## Резюме

Прогон #2 (tcalc) показал существенное улучшение по сравнению с прогоном #1 (orch): стоимость на задачу упала с ~$17 до ~$5, rework с 1 до 0, время с 27 до 18 минут. Ключевые факторы: строгие фазы, сигнальные файлы, interface-файл между фазами, конкретика в description. Однако код tcalc содержит несколько багов, тесты пропускают критические сценарии, а архитектура оркестратора имеет фундаментальные ограничения по стоимости (baseline cost воркера ~$3 делает систему неэффективной для задач меньше moderate).

## Код tcalc

### Корректность: 6/10 -- обнаружены баги

**Баг 1: `resolveProjectDir()` -- неочевидная логика маппинга путей**
Файл: `/Users/user/orchestra/output/token-calc/bin/tcalc.js:30`
```js
const projectName = cwd.replace(/\//g, '-');
```
Для CWD `/Users/user/orchestra` результат будет `-Users-danilasartakov-orchestra`. Ведущий дефис -- побочный эффект замены первого `/`. В session-analysis.md написано: `projectId: /Users/x/orchestra -> -Users-x-orchestra` -- то есть с дефисом. Если реальные директории в `~/.claude/projects/` именуются иначе (без ведущего дефиса), утилита не найдёт проект. Хрупкая зависимость от внутреннего формата Claude Code без документации.

**Баг 2: `--since` фильтрует по `lastTimestamp`, а не по `firstTimestamp`**
Файл: `/Users/user/orchestra/output/token-calc/bin/tcalc.js:86-89`
```js
filtered = sessions.filter(s => {
  if (!s.lastTimestamp) return false;
  return new Date(s.lastTimestamp) >= sinceDate;
});
```
Сессия, начатая задолго до `--since`, но с последним сообщением после -- будет включена целиком (все токены, включая до since). Сессия Мозга, запущенная за часы до прогона, попадёт в подсчёт и завысит стоимость. Это может быть намеренным, но не документировано и не тестируется.

**Баг 3: Нет `--help` и обработки невалидных аргументов**
Файл: `/Users/user/orchestra/output/token-calc/bin/tcalc.js:11-26`
`tcalc --since` (без значения) -- silent undefined. `tcalc --foo` -- молча игнорируется. Нет `--help` / `--version`.

**Баг 4: `formatReport` с `breakdown !== false` по умолчанию включает breakdown**
Файл: `/Users/user/orchestra/output/token-calc/src/formatter.js:23`
```js
const breakdown = options.breakdown !== false;
```
`formatReport(sessions)` или `formatReport(sessions, {})` -- breakdown включится (undefined !== false = true). Противоречит task-004: "Без --breakdown: показать только TOTAL". В tcalc.js дефолт `breakdown: false`, так что CLI работает правильно. Но API модуля -- нелогичен: если вызывать formatReport напрямую без options, поведение обратное ожидаемому.

**Баг 5: `findSessions` сортирует лексикографически**
Файл: `/Users/user/orchestra/output/token-calc/src/parser.js:12`
```js
.sort();
```
Сортировка по имени файла, не по дате. Если имена не содержат хронологический компонент, порядок в отчёте произвольный.

**Баг 6: Нет обработки ошибок при чтении JSONL**
Файл: `/Users/user/orchestra/output/token-calc/src/parser.js:27-28`
Если файл заблокирован или нет прав на чтение -- `fs.createReadStream` выбросит ошибку, которая не обработана в parseSession. Попадёт в `main().catch()` в tcalc.js, но с невнятным сообщением.

### Тесты: 5/10 -- пропущены критические сценарии

**Покрыто хорошо:**
- happy path для parser, pricing, formatter
- edge cases: null, undefined, NaN для formatNumber/formatCost
- malformed JSON в parser
- fallback моделей в pricing
- пустые файлы, отсутствие usage

**Пропущено:**

1. **Нет тестов для `bin/tcalc.js`** -- CLI entry point (parseArgs, resolveProjectDir, main flow, --since фильтрация) не тестируется. Это самый сложный и багоопасный модуль.

2. **Нет теста `formatReport` с `breakdown: false`** -- есть тест с `breakdown: true` (строка 193), но нет проверки что при `false` строки сессий НЕ выводятся.

3. **Нет теста `formatTimestamp`** -- функция определена в formatter.js:15, не экспортируется, не тестируется.

4. **Нет теста на суммирование totalCost в TOTAL** -- тест `sums totals across multiple sessions` проверяет наличие подстрок "3,000" и "$4.50" в output, но не проверяет правильность суммирования cost по всем сессиям отдельно от токенов.

5. **Нет теста для нескольких моделей в одном отчёте** -- что если одна сессия opus, другая sonnet? formatReport не пересчитывает стоимость, он берёт `s.cost.totalCost` -- это правильно, но нет теста.

6. **Нет негативных тестов** -- parseSession с несуществующим файлом, findSessions с файлом (не директорией).

### Архитектура: 7/10

**Плюсы:**
- Чистое разделение: parser/pricing/formatter -- одна ответственность на модуль
- Zero dependencies (только встроенные модули Node.js)
- Streaming parsing через readline (не загружает весь файл в память)
- Fallback-механизм для неизвестных моделей

**Минусы:**
- `formatTimestamp`, `resolvePricing` не экспортируются -- нельзя протестировать отдельно
- `resolveProjectDir`, `parseArgs` живут в bin/tcalc.js -- нельзя протестировать без запуска CLI
- Нет типизации (JSDoc хотя бы) -- интерфейсы описаны только в phase-1-interface.md
- package.json не содержит scripts.test -- нельзя запустить `npm test`

## Что сработало в v2

### 1. Строгие фазы (главная оптимизация)
В v1 тестер и writer запущены одновременно с кодером -- тестер 10 минут поллил несуществующие файлы, потом завис, нужен restart. В v2: Phase 1 (3 кодера параллельно) -> ревью -> Phase 2 (CLI + тесты). Ноль idle-поллинга, ноль рестартов. Экономия: ~$10-15 + 10 минут.

### 2. Сигнальные файлы вместо JSON polling
В v1: Мозг каждые 15 сек читал все task-*.json (~150 tool calls за сессию). В v2: `ls .brain/signals/` + `check-workers.sh` = 2 calls/цикл. При adaptive polling (3 мин пауза) -- воркеры закончили за паузу, 0 лишних циклов.

### 3. Interface-файл между фазами
`phase-1-interface.md` содержит точные сигнатуры всех модулей. Воркеры Phase 2 знали API без чтения чужого кода. Результат: 0 rework на интеграционных задачах.

### 4. Конкретика в description задач
task-002 содержит точные цены, формулу, имена полей. task-004 -- пошаговый алгоритм. Результат: 0 rework на всех 5 задачах (кроме task-003 -- см. ниже).

### 5. Компактные промпты (50 -> 10-17 строк)
Протокол вынесен в WORKER_PROTOCOL.md. Экономия ~200 строк system prompt суммарно.

### 6. `--dangerously-skip-permissions` в run-worker.sh
Решает проблему v1, где тестер не мог запустить `node --test`.

## Что НЕ сработало / можно улучшить

### 1. Мозг доминирует по стоимости
~$28 из ~$52 = 54% бюджета -- координация. CLAUDE.md ~600 строк, загружается как system prompt каждого запроса Мозга. Стоимость Мозга фиксирована и не масштабируется с количеством задач.

### 2. task-003 (formatter) -- description в одну строку
```json
"description": "Output formatter module"
```
task-001 и task-002 содержат полный 5C формат. task-003 -- одна строка. Нарушает собственный принцип "5C PROMPTS". Результат оказался рабочим, но это удача, а не методология.

### 3. Все 5 задач в JSON остались в статусе "review"
`/Users/user/orchestra/.brain/tasks/task-001.json:8` и аналогично task-002 - task-005. state.json показывает `done`, metrics.jsonl -- `done`, но JSON задач -- `review`. Cold restart сломается: Мозг увидит 5 задач в review и попытается ревьюить заново.

### 4. Timestamps рассинхронизированы
state.json: `started_at: "2026-03-01T12:00:00Z"`. brain.log: `[2026-03-01T03:44:37Z] CLASSIFY`. Разница 8+ часов. state.json написан с округлёнными/неточными таймстампами.

### 5. metrics.jsonl не содержит duration_sec
Формат из CLAUDE.md требует `duration_sec`, но в реальных записях поле отсутствует. Невозможно анализировать время выполнения задач.

### 6. Все 5 метрик записаны с одним timestamp
`"timestamp":"2026-03-01T04:02:13Z"` для всех. Phase 1 завершилась в 03:48, Phase 2 в 04:02. Записаны скопом.

### 7. Изоляция через промпт, не технически
`--dangerously-skip-permissions` + "НЕ трогай файлы вне owned_files" в промпте. Воркер технически может изменить любой файл.

### 8. Мозг не запускал тесты после ревью
brain.log v2 не содержит записей о запуске тестов. В v1 Мозг запускал тесты сам. В v2 acceptance criteria task-005 содержит "node --test проходит без ошибок", но нет доказательств что Мозг это верифицировал.

## Архитектурные рекомендации для v3

### 1. Shell-оркестратор для simple/moderate задач
Мозг ($28 фиксированно) = overhead 112% для $25 работы. Альтернатива: bash/python скрипт, который:
- Читает plan.yaml (Мозг генерирует один раз -> $3-5)
- Запускает воркеров по фазам (shell = $0)
- Проверяет signals + exit codes (shell = $0)
- Вызывает Мозга только для ревью ($5-7)
- Итого: $8-12 вместо $52

### 2. Техническая изоляция воркеров
Git worktree для каждого воркера или Docker с mount только owned_files. Убирает зависимость от "честности" LLM.

### 3. Schema validation для задач
JSON Schema / zod для task, state, worker файлов. Валидировать перед записью. Ловить ситуации типа task-003 с пустым description.

### 4. Встраивание interface в description вместо ссылки на файл
phase-1-interface.md -- 20 строк. Два Read tool call для чтения (interface + src/) = ~$0.50 на воркера. Встроить в description = 0 tool calls.

### 5. Budget guard
Перед запуском воркера -- оценить стоимость. Если суммарно > лимит -- спросить человека. Записывать реальную стоимость в workers/*.json.

## Рекомендации по промптам

### 1. Валидация description перед записью
task-003 не содержит 5C формата. Мозг должен проверять: description > 50 символов, содержит секции КОНТЕКСТ/ЗАДАЧА/ОГРАНИЧЕНИЯ.

### 2. Пример вызова в каждом description
task-001 описывает API, но не показывает пример: `const result = await parseSession('/path/to/file.jsonl')`. task-004 содержит пошаговый алгоритм -- это лучше.

### 3. Мозг должен запускать верификацию после accept
Если acceptance criteria содержит "node --test проходит" или "node bin/tcalc.js выводит отчёт" -- Мозг обязан запустить команду и проверить exit code + output.

### 4. Встроить interface в description Phase 2
Убрать "Прочитай .brain/results/phase-1-interface.md" из промптов. Вместо этого -- вставить 20 строк interface прямо в description задач Phase 2. Экономия 2 Read tool calls на воркера.

### 5. Добавить секцию "Ошибки" в WORKER_PROTOCOL.md
Что делать при: ошибка зависимости, файл уже существует, тесты не проходят.

### 6. Специализацию ("Делай ровно то что просят") вынести в WORKER_PROTOCOL.md
Сейчас дублируется во всех промптах кодеров. Оставить в промпте только уникальное: язык, зависимости, стиль.
