# Аудит системы Orchestra — прогоны #1 и #2

Ты — независимый аудитор AI-оркестратора Orchestra. Система управляет командой воркеров (Claude Code процессов) через tmux для параллельного выполнения задач.

Твоя задача — **полный аудит** двух прогонов системы и рекомендации по улучшению.

---

## Контекст

Orchestra = "Мозг" (основной Claude Code) + N воркеров (дочерние Claude Code в tmux).
- Мозг: классифицирует задачу, декомпозирует, генерирует промпты, спавнит воркеров, мониторит, ревьюит результаты
- Воркеры: читают задачу из JSON, выполняют, пишут результат, ставят сигнал

**Прогон #1** (v1): CLI-утилита `orch` (мониторинг). 3 воркера, 1 фаза, ~27 мин, **~$52**
**Прогон #2** (v2, с оптимизациями): CLI-утилита `tcalc` (подсчёт токенов). 5 воркеров, 2 фазы, ~18 мин, **стоимость замерить**

---

## Что аудитировать

### 1. Архитектура системы
Прочитай `CLAUDE.md` (system prompt Мозга) и оцени:
- Полнота протокола: все ли edge cases покрыты?
- Масштабируемость: что сломается при 10+ воркерах? При 5+ фазах?
- Единые точки отказа: что если Мозг упадёт? Воркер зависнет?
- Cold restart: реально ли state.json достаточен для продолжения?

### 2. Качество кода (tcalc)
Прочитай все файлы в `output/token-calc/` и оцени:
- Корректность: правильно ли считает? Есть ли баги?
- Edge cases: пустые файлы, невалидный JSON, неизвестные модели
- Тесты: покрытие, качество assertions, пропущенные сценарии
- Архитектура: разделение ответственности, интерфейсы между модулями

### 3. Эффективность процесса
Сравни прогон #1 и #2:
- Стоимость в токенах (замерь через `node output/token-calc/bin/tcalc.js --since "2026-03-01T03:40:00Z" --breakdown`)
- Количество rework (v1: 1 restart тестера, v2: 0 rework)
- Время выполнения
- Оверхед координации vs реальная работа

### 4. Оптимизации v2 — сработали?
Прочитай `memory/patterns.md` и `memory/anti-patterns.md`, проверь:
- Протокол в файле (WORKER_PROTOCOL.md) vs inline в промпте — реально сэкономило?
- Сигнальные файлы vs polling JSON — разница в tool calls?
- Adaptive polling (3 мин пауза) — Мозг не пропускал завершения?
- Строгие фазы vs запуск всех сразу — влияние на idle time воркеров?
- Компактные промпты (10 строк vs 50) — качество результата не упало?

### 5. Промпты воркеров
Прочитай промпты в `.brain/prompts/` и задачи в `.brain/tasks/`:
- 5C формула (Context, Command, Constraints, Criteria, Completion) — применена?
- Бинарные acceptance criteria — можно ли проверить за <30 сек?
- Конкретика: примеры данных, ожидаемый формат, формулы — достаточно?
- Что можно улучшить?

### 6. Слабые места и риски
- Какие задачи воркеры выполнили бы неправильно без подробного description?
- Где система хрупкая (один неверный JSON — всё ломается)?
- Что не масштабируется?

## Важно

- Будь конкретным. "Код хороший" — бесполезно. "parser.js:58 — сравнение строк дат может дать неверный результат для разных таймзон" — полезно.
- Считай в долларах, не в абстракциях.
- Если нашёл баг — покажи конкретный input и ожидаемый vs реальный output.
- Не хвали, анализируй. Цель — найти что улучшить.
