Ты получил лог первого реального прогона системы Orchestra — AI-оркестратора, который управляет командой воркеров (Claude Code процессов) через tmux. Проанализируй этот прогон и предложи оптимизации.

### Что нужно проанализировать:
1. **Расход токенов** — где можно сэкономить? ($52.03 на простую CLI-утилиту — это много или норм?)
2. **Эффективность промптов воркеров** — слишком длинные? Лишняя информация?
3. **Мониторинг Мозга** — Мозг каждые 15 секунд читал все JSON-файлы для проверки статусов. Это генерирует много tool calls и токенов. Как оптимизировать?
4. **Параллелизм** — task-002 (тесты) и task-003 (README) зависели от task-001 (код). Можно ли было организовать лучше?
5. **Heartbeat-протокол** — воркеры каждые 30 сек обновляли JSON через jq. Есть ли способ проще?
6. **Перезапуск worker-02** — тестер завис при первом запуске, пришлось рестартить. Почему? Как предотвратить?
7. **Архитектурные улучшения** — что изменить в протоколе взаимодействия Мозг ↔ воркеры?

## Ошибки и инциденты

### 1. CLAUDECODE env var (критическая)
```
Error: Claude Code cannot be launched inside another Claude Code session.
Nested sessions share runtime resources and will crash all active sessions.
```
**Причина:** Мозг запущен как Claude Code сессия, spawn-worker.sh запускает ещё один `claude` внутри tmux. Переменная CLAUDECODE наследуется.
**Фикс:** Добавлена строка `unset CLAUDECODE` в run-worker.sh перед `exec claude`.
**Потеря:** ~3 минуты на диагностику и фикс.

### 2. worker-02 (tester) завис
**Симптом:** После второго запуска worker-02 не подхватил задачу ~11 минут.
**Вероятная причина:** Тестер зависел от task-001, но в description было написано "подожди пока coder создаст файлы", а не чёткий механизм ожидания. Воркер мог зациклиться проверяя несуществующие файлы.
**Фикс:** Принудительный перезапуск через kill-worker + spawn-worker.
**Потеря:** ~10 минут + дублирование начального промпта (доп. токены).

### 3. worker-02 не запустил тесты
**Симптом:** Тестер написал 56 тестов, но не смог запустить `node --test` из-за отсутствия разрешения в allowedTools.
**Причина:** В spawn-worker.sh allowedTools содержит: `Bash(command)`, `Read`, `Write`, `Edit`, `Glob`, `Grep`. Bash разрешён, но claude запрашивает отдельное одобрение для конкретных команд — и в режиме `-p` (pipe, неинтерактивный) нет возможности одобрить.
**Потенциальный фикс:** Добавить `--dangerously-skip-permissions` для неинтерактивных воркеров ИЛИ предоставить конкретные разрешения в командной строке.

### Промпты воркеров

Каждый промпт ~50 строк, структура одинаковая:
1. Идентификация (роль, ID)
2. Как найти задачу (scan .brain/tasks/)
3. Протокол работы (status updates, heartbeat)
4. Правила (не трогать чужие файлы)
5. Специализация (4-5 строк специфичных для роли)

**Наблюдение:** Промпт содержит много boilerplate (heartbeat инструкции, jq команды, правила). Только ~10% промпта — специфика задачи.

---

### Скрипты

#### spawn-worker.sh
- Создаёт tmux сессию/окно
- Запускает run-worker.sh внутри окна
- Регистрирует воркера в workers/*.json

#### run-worker.sh
- Читает промпт из .brain/prompts/
- Запускает `claude --system-prompt -p` с ограниченными allowedTools
- `unset CLAUDECODE` (фикс для nested sessions)
- Передаёт initial task через `-p`

### Мониторинг Мозга

Мозг (основной Claude Code процесс) мониторил воркеров циклически:
- Читал все `task-*.json` каждые ~15-30 секунд
- Проверял heartbeat воркеров
- Проверял .brain/results/ на новые файлы
- Каждый цикл = несколько Read tool calls + bash jq

**Оценка:** ~30-40 циклов мониторинга за 27 минут. Каждый цикл = 3-6 tool calls = значительный расход токенов Мозга (возможно 30-50% от общей стоимости).

### Token-tracker (виджет)

Виджет считал токены так:
1. Путь к проекту → projectId: `/Users/x/orchestra` → `-Users-x-orchestra`
2. Директория JSONL: `~/.claude/projects/{projectId}/*.jsonl`
3. Фильтр: только файлы с birthtime >= state.json.started_at
4. Парсинг: строки с `"type":"assistant"` → message.usage → pricing по модели
5. Итого: inputTokens, outputTokens, cache*, totalCost

**Ограничение:** Учитывает только сессии воркеров, не учитывает токены самого Мозга (он был запущен до started_at).

## Вопросы для анализа

1. **$52 за CLI-утилиту из 4 файлов + тесты + README.** Сколько из этого — мониторинг Мозга? Можно ли снизить до $10-15?
2. **Мониторинг через Read tool calls** — можно ли заменить на event-driven подход (воркеры пишут в один файл, Мозг делает `tail -f`)?
3. **depends_on реализован через polling** — воркер-тестер каждые 30 сек проверяет наличие файлов. Можно ли передать зависимости через фазовый механизм (не запускать тестера до завершения кодера)?
4. **Промпты** — 90% boilerplate, 10% задача. Можно ли вынести общий протокол в один файл и импортировать?
5. **allowedTools** — тестер не смог запустить тесты. Нужен ли `--dangerously-skip-permissions` или можно точнее настроить permissions?
6. **Heartbeat через jq** — каждые 30 сек каждый воркер делает jq + mv. Это 3 воркера × 2 tool calls × 50+ раз = 300+ лишних tool calls. Нужен ли heartbeat вообще?
7. **Виджет не считает токены Мозга** — как включить? Сессия Мозга запущена до started_at.
